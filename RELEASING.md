# Releasing

AllureDeck uses [semantic-release](https://semantic-release.gitbook.io/) for fully automated versioning and releases based on [Conventional Commits](https://www.conventionalcommits.org/).

## Version Scheme

[Semantic Versioning 2.0.0](https://semver.org/) — `MAJOR.MINOR.PATCH`

| Tag format      | Scope                        |
|-----------------|------------------------------|
| `vX.Y.Z`       | Application (API + UI)       |
| `helm-vX.Y.Z`  | Helm chart (independent)     |

## What Triggers a Release

Commit messages on `main` determine the version bump:

| Commit prefix                              | Bump  | Example                                      |
|--------------------------------------------|-------|----------------------------------------------|
| `feat:`                                    | Minor | `feat: add project comparison view`          |
| `fix:`                                     | Patch | `fix: correct date parsing for Allure 3`     |
| `perf:`                                    | Patch | `perf: reduce report summary query time`     |
| `revert:`                                  | Patch | `revert: undo handler refactor`              |
| `BREAKING CHANGE` footer or `!` suffix     | Major | `feat!: redesign REST API endpoints`         |

### Commits That Do NOT Trigger a Release

| Commit prefix    | Reason                                             |
|------------------|----------------------------------------------------|
| `chore(deps):`   | Dependabot dependency updates                      |
| `chore(docker):` | Dependabot Docker base image updates               |
| `chore(ci):`     | Dependabot GitHub Actions updates                  |
| `chore(release):`| Auto-generated by semantic-release (`[skip ci]`)   |
| `docs:`, `style:`, `test:`, `build:`, `ci:` | Non-releasable by convention         |

## Application Release (automatic)

Push to `main` with a conventional commit message:

```bash
git commit -m "feat: add new feature"
git push origin main
```

```text
push to main
    |
    v
[CI] Run tests (API + UI)
    |
    v
[semantic-release] Analyze commits since last tag
    |
    +-- No releasable commits --> stop
    |
    v
Create vX.Y.Z tag
    |
    v
Update CHANGELOG.md + commit
    |
    v
Create GitHub Release with notes
    |
    v
Build multi-arch Docker images (amd64 + arm64)
    |
    v
Push to ghcr.io/mkutlak/alluredeck-api
Push to ghcr.io/mkutlak/alluredeck-ui
    |
    v
Attest build provenance
```

## Helm Chart Release

The Helm chart version is **independent** from the application version. App releases do **not** automatically update or publish the Helm chart. After an app release, you must manually update the chart to reference the new version:

```bash
# After app release vX.Y.Z, update the chart's appVersion
yq -i '.appVersion = "X.Y.Z"' charts/alluredeck/Chart.yaml

# Optionally bump the chart version too
make helm-release              # patch bump

git add charts/ && git commit -m "chore: bump chart appVersion to X.Y.Z"
git push origin main
```

Chart releases are triggered by changes to `charts/**` on `main`.

For chart-only changes (templates, values, defaults):

```bash
# Edit chart files
vim charts/alluredeck/...

# Bump chart version and commit
make helm-release              # patch: 0.1.0 → 0.1.1
make helm-release BUMP=minor   # minor: 0.1.0 → 0.2.0
make helm-release BUMP=major   # major: 0.1.0 → 1.0.0

git push origin main
```

```text
push to main (charts/** changed)
    |
    v
Read version from Chart.yaml
    |
    v
Check if helm-vX.Y.Z tag exists
    |
    +-- Tag exists --> skip (idempotent)
    |
    v
Package chart + push to oci://ghcr.io/mkutlak/charts
    |
    v
Create helm-vX.Y.Z git tag
```


## Dependabot Policy

Dependabot PRs use scoped `chore()` prefixes that are excluded from triggering releases:

- Go/npm deps: `chore(deps): ...`
- GitHub Actions: `chore(ci): ...`
- Docker images: `chore(docker): ...`

**Workflow**: Dependabot PRs are merged individually (auto-merge OK), but they won't create releases. When you want a release, create a `feat:` or `fix:` commit.

## Branch Protection

### Main branch (`main`)

- No force pushes
- No deletions
- Required status checks: `Run tests` (not enforced on branch creation)
- Admin bypass for repository owner

### Tags (`v*`, `helm-v*`)

- No deletion
- No force-updating
- GitHub Actions (via `GITHUB_TOKEN`) can create tags
- Manual tag creation is blocked

## Manual Release Override

If you need to force a specific version bump, use the commit message footer:

```text
fix: update error handling

BREAKING CHANGE: error response format changed from string to object
```

This forces a major bump regardless of the `fix:` prefix.

To skip CI entirely on a commit:

```text
chore: update docs [skip ci]
```

## Prerequisites

- GitHub Pages enabled on `gh-pages` branch (Settings → Pages)
- `GITHUB_TOKEN` has `contents: write` and `pages: write` permissions
