// Quick Start

Install

Pull and install from the OCI registry. No helm repo add needed.

Pull the chart
$ helm pull oci://ghcr.io/mkutlak/charts/alluredeck
Install to cluster
$ helm install alluredeck oci://ghcr.io/mkutlak/charts/alluredeck
Install with custom values
$ helm install alluredeck oci://ghcr.io/mkutlak/charts/alluredeck \ --set api.ingress.enabled=true \ --set api.ingress.host=allure.example.com \ --set ui.ingress.enabled=true \ --set ui.ingress.host=allure.example.com

// Capabilities

Key Features

Production-ready defaults. Secure by design.

💾

Flexible Storage

Local persistent volumes for on-prem or S3-compatible storage (AWS, MinIO) for cloud deployments. IRSA support for EKS.

🔒

Built-in Auth

Role-based access with admin and viewer roles. JWT authentication with auto-generated signing keys and credentials.

🛡️

Network Policies

Optional Kubernetes NetworkPolicies to restrict pod-to-pod communication and enforce least-privilege networking.

🔐

Auto-generated Secrets

Secure credentials generated on first install and preserved across upgrades. Supports external secret managers.

🧱

Security Hardened

Non-root containers, read-only root filesystems, dropped capabilities, and restricted security contexts by default.

📊

Health Monitoring

Liveness, readiness, and startup probes for both API and UI. Resource limits with Go-specific GOMEMLIMIT tuning.

// System Design

Architecture

Two-component deployment: API handles report processing, UI serves the dashboard.

Ingress
allure.example.com
→
Frontend
React UI
→
Backend
Go API
→
Storage
PVC / S3

Each component runs as a separate Deployment with dedicated Service, ServiceAccount, and optional NetworkPolicy. The UI proxies API requests, and CORS is auto-configured when ingress is enabled.
